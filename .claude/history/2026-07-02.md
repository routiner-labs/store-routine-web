# 2026-07-02 작업 기록

## 89. 문서함 신규 기능 (업무 카테고리, 에디터 적용, 목업 30건)

사용자 요청: "문서함" 개념을 업무 카테고리에 추가하고, 에디터를 적용. 목업 데이터 약 30건 포함. 최초에는 좌/우 분할(리스트|상세) 레이아웃으로 만들었으나, 사용자 피드백으로 리스트 → 클릭 시 상세 페이지 이동 방식으로 전면 재구성.

추가된 파일:
- `src/mock/documents.ts` — `DocumentCategory`(8종: 포스·결제/고객 응대/청소·시설/재고·보관/안전·비상/매장 운영 규칙/이벤트·마케팅/기타), `StoreDocument` 타입, `DOCUMENT_CATALOG` 목업 30건(카테고리별 실제 업무 매뉴얼 성격의 HTML 콘텐츠 — `요청함`/`테스크` 목업과 동일하게 타입+데이터를 한 파일에 colocate)
- `src/app/owner/documents/page.tsx` — 목록 페이지(검색, 카테고리 필터 칩, 문서 카드 리스트). 행 클릭 시 `/owner/documents/[id]`로 이동, "새 문서" 버튼은 `/owner/documents/new`로 이동
- `src/app/owner/documents/page.module.css`
- `src/app/owner/documents/[id]/page.tsx` — 상세 페이지. 보기 모드(렌더된 HTML, 작성자·작성일/수정일, 수정/삭제 아이콘 버튼)와 수정 모드(`DocumentForm` 재사용) 전환. 삭제는 `useConfirm()` 확인 후 처리, 성공 시 `useToast()` + 목록으로 이동(요청함 `[id]` 페이지와 동일 패턴 — 목업이라 삭제/수정이 `DOCUMENT_CATALOG` 원본을 변형하지 않고 로컬 상태만 갱신, 요청함과 동일한 한계 유지)
- `src/app/owner/documents/[id]/page.module.css`
- `src/app/owner/documents/new/page.tsx` — 신규 작성 페이지. `DocumentForm` 사용, 저장 시 토스트 후 목록으로 이동
- `src/app/owner/documents/new/page.module.css`
- `src/app/owner/documents/DocumentForm.tsx` — 제목 입력 + 카테고리 select + `RichTextEditor`(기존 `src/components/RichTextEditor` 재사용, Lexical 기반 — 새 에디터 라이브러리 도입 없음) + 저장/취소(+선택적 삭제) 폼. 상세(수정)와 신규 작성 페이지에서 공통 사용
- `src/app/owner/documents/DocumentForm.module.css`

수정된 파일:
- `src/app/owner/OwnerNav.tsx` — `업무` 카테고리 그룹에 `문서함`(`/owner/documents`, `LiaFolderOpenSolid`) 항목 추가(업무리스트·요청함 다음)

주요 결정:
- 에디터는 신규 도입 없이 기존 `src/components/RichTextEditor`(Lexical, `initialHtml`/`onChange` HTML 문자열 인터페이스) 그대로 재사용
- 최초 시도한 좌/우 분할(마스터-디테일) 레이아웃은 사용자 피드백으로 폐기 → 요청함(`requests`/`requests/[id]`)과 동일한 목록+상세 라우트 분리 구조로 재작성
- 좌/우 분할 버전에서 상세/폼을 `renderForm()` 형태로 JSX 내 직접 호출했다가 `react-hooks/refs`, `react-hooks/purity` ESLint 에러(ref를 렌더 중 호출된 함수 안에서 읽음) 발생 → 별도 `DocumentForm` 컴포넌트로 분리해 해결(체크리스트 페이지의 `initialHtml={methodHtmlRef.current}` 직접 JSX 참조 패턴과 동일한 원리)
- 카테고리 뱃지는 기존 요청함의 `.typeTag` 사각형 뱃지 스펙(6px radius, 11px, 3px 8px padding)을 그대로 따름
- 목업 데이터라 생성/수정/삭제는 `DOCUMENT_CATALOG` 원본을 변경하지 않음(요청함 `[id]` 페이지의 기존 동작과 동일한 한계) — 실제 영속화가 필요해지면 별도 백엔드/상태관리 도입 필요
- 검증: `tsc --noEmit`·`eslint` 클린, dev 서버에서 목록/상세(doc1·doc15·doc30)/신규/미존재 id 라우트 모두 curl로 SSR HTML 확인(문서 30건, 카테고리 칩, 본문 렌더링 정상), 사이드바에 `문서함` 노출 확인. 헤드리스 브라우저 도구(chromium-cli/playwright) 미설치로 실제 클릭 인터랙션은 브라우저 시각 확인 대신 SSR 출력 검증으로 대체 — 필요 시 `/run-skill-generator`로 브라우저 검증 스킬 구축 권장
